## 1. 概述

本次维护针对 GateServer 进行了深度的代码审查，主要修复了潜在的资源泄漏、并发安全隐患以及协议处理的健壮性问题。重点改进了数据库连接池的生命周期管理（RAII），增强了 HTTP 请求的处理逻辑，并优化了线程池的启动顺序。

## 2. 详细修复内容

### 2.1 MySQL 连接池 (MysqlPool & MysqlDao)

#### 问题分析
1.  **资源泄漏风险**：原有的 `getConnection()` 返回 `std::unique_ptr`，依赖调用者手动执行 `returnConnection()`。如果在业务逻辑中发生异常（Exception）或提前返回，连接将无法归还，导致连接池逐渐耗尽。
2.  **僵尸连接问题**：从池中获取连接时未检查连接的有效性。如果 MySQL 服务重启或连接超时断开，服务层会获取到无效连接从而报错。

#### 优化方案
*   **引入 RAII (资源获取即初始化)**：修改 `getConnection()` 返回 `std::shared_ptr<sql::Connection>`，并绑定自定义删除器（Deleter）。
    *   当 `shared_ptr` 引用计数归零析构时，会自动调用 `returnConnection` 将连接归还给池。
    *   **效果**：`MysqlDao` 层不再需要手动归还连接，代码更加简洁且异常安全。
*   **自动重连机制**：在获取连接时增加 `isValid()` 检查。如果连接失效，自动尝试重新建立连接，提高了系统的自愈能力。

**代码变更**:
- `include/MysqlPool.h`: 修改 `getConnection` 签名及实现。
- `src/MysqlDao.cpp`: 移除所有显式的 `pool_->returnConnection` 调用。

### 2.2 HTTP 连接处理 (HttpConnection)

#### 问题分析
1.  **请求挂起 (Hang)**：原逻辑仅处理 `GET` 和 `POST` 方法。如果客户端发送 `HEAD`, `PUT` 或其他方法的请求，服务器不发送任何响应直接返回，导致客户端连接一直挂起直到超时。
2.  **内存越界崩溃风险**：`UrlDecode` 函数中使用 `assert` 检查 URL 编码格式。在 Release 模式下 `assert` 可能失效，而在 Debug 模式下遇到恶意构造的 URL（如 `%` 后无足够字符）会导致服务直接 Crash。

#### 优化方案
*   **兜底响应**：在 `HandleRequest` 末尾增加对未知方法的默认处理，返回 `405 Method Not Allowed`，确保所有请求都有明确的响应。
*   **安全解码**：将 `assert` 替换为运行时的边界检查。遇到非法编码格式时进行降级处理（原样保留或安全截断），防止服务崩溃。

**代码变更**:
- `src/HttpConnection.cpp`: 增加 `405` 响应逻辑；重构 `UrlDecode` 安全性。

### 2.3 IO 线程池 (AsioIOServicePool)

#### 问题分析
*   **启动顺序竞争**：原构造函数在同一个循环中创建 `IOService` 并启动线程。理论上可能存在 `work_guard` 还未完全建立，线程就已经跑飞的情况（虽然概率较低，但在高并发初始化时是不安全模式）。

#### 优化方案
*   **两阶段初始化**：将资源的创建（`io_context`, `work_guard`）与线程的启动（`thread`）分离。确保所有上下文资源准备就绪后，再统一启动工作线程。

**代码变更**:
- `src/AsioIOServicePool.cpp`: 分离循环逻辑。
